## Phase Details

### Phase 19: callLLM 重构 + GLM-4.6V 升级
**Goal**: 视觉评价链路可信 — GLM-4.6V 通过 multimodal content blocks 真实打分,消除 v2.0 bracketed-path prompt 静默失效风险
**Depends on**: Nothing (BLOCKER,先做)
**Requirements**: D1-01, D1-02, D1-03, D1-04
**Success Criteria** (what must be TRUE):
  1. `callLLM`/`callLLMJson` 接受并正确序列化 `{type:'image_url', image_url:{url}}` content block(不再走文本嵌入路径),单元测试覆盖 multimodal payload 构造
  2. 全 codebase 模型名统一到 `ZHIPU_VISION_MODEL` env var(消除 5 处硬编码跨 3 个版本),`continuity-auditor.js:398` 不再含 `glm-4v-flash` 字面量
  3. 50-pair golden set baseline 报告产出(升级前后 score 分布 CSV/JSON 对比),thinking 模式对评分确定性影响已记录
  4. `_scoreCache` 按 model_version 命名空间失效,旧 cache key 不会污染新模型评分(单测覆盖版本切换场景)
  5. 降级契约:GLM-4.6V 不可达时 callLLM 返回 null(保留 v2.0 quality-gate null-fallback 语义),degraded E2E 仍 <5s(DEGRADE-01/02)
**Plans**: TBD

### Phase 20: AssetBus Schema 扩展
**Goal**: 集成脊柱就位 — 所有 v3.0 新模块通过 typed slot 持久化,获得原子写入 + cache + git checkpoint 能力
**Depends on**: Phase 19
**Requirements**: SCHEMA-01, SCHEMA-02, SCHEMA-03
**Success Criteria** (what must be TRUE):
  1. `ASSET_SCHEMA` 新增 3 typed slots:`creative-history`、`failed-shots`、`finetune-dataset`,每个 slot 有明确 schema 定义(shots/edges/version 等)
  2. v2.0 旧数据在缺失新 slot 时可读(向后兼容测试通过,envelope `{value, derived_from, content_hash}` 字段缺失走 null fallback)
  3. 跨 phase 写入是原子的(并发写入测试不产生 corruption),cache 在写入后失效(读取到新值)
  4. 降级契约:AssetBus 写入失败时记录 error 并继续(slot 不可用不阻断 pipeline),degraded E2E 仍 <5s(DEGRADE-01/02)
**Plans**: TBD

### Phase 21: BlacklistEngine + bad case 持久化
**Goal**: 已知坏模式不再消耗 GPU 预算 — 跨 run 累积的 failed_shots 库 + 语义匹配拦截,cloud-production 提交前过滤
**Depends on**: Phase 20
**Requirements**: B5-01, B5-02, B5-03, B5-04, B5-05, B5-06
**Success Criteria** (what must be TRUE):
  1. `failed_shots.json` 跨 run 累积(多次 pipeline run 后文件包含所有历史 failure,非覆盖写),启动时加载到内存
  2. `lib/blacklist-engine.js` 实现语义匹配(GLM-4.6v embedding 相似度 ≥0.92,非 regex 子串),单元测试覆盖 "violins vs violence" 类误匹配防御
  3. `cloud-production` before-hook 对命中 shot 走 skip 语义(非 retry、非 fail,ShotParallelScheduler 返回值可区分三种状态)
  4. Escape hatch(`config.blacklist.disabled: true`)紧急关闭生效;TTL 默认 30 天过期 stale 条目自动清理;操作日志(谁/何时/为何)可审计
  5. 降级契约:BlacklistEngine 不可用时默认 allow(generation 不被阻断),degraded E2E 仍 <5s,单元测试覆盖 GLM 不可达路径(DEGRADE-01/02)
**Plans**: TBD

### Phase 22: Seedance 2.0 Audio-Visual Sync
**Goal**: 原生音画同步接入 — Seedance 2.0 通过 audio_refs + @Audio1 prompt 绑定生成口型同步视频,中文场景达到现实阈值
**Depends on**: Phase 19, Phase 21
**Requirements**: A2-01, A2-02, A2-03, A2-04, A2-05
**Success Criteria** (what must be TRUE):
  1. `getOmniReferencePack()` 接受 `audioRefs` 参数,产出 prompt 中含 `@Audio1` token 绑定(参数缺失或空时不绑定,非静默失败)
  2. `cloud-production` handler 实现 voice → cloud-production 时序锁(voice phase 未完成时不提交 Seedance 任务),时序违反时显式报错
  3. Seedance 任务提交时强制校验:audio_refs 非空但 prompt 无 `@Audio` token → reject(避免 Pitfalls 陷阱 1 静默失败)
  4. 中文 lip sync 测试集(≥10 样本,douyin 平台短剧场景)产出口型质量报告;`lip_sync_threshold` 从 1.0 调整为基于测试集的现实阈值(预计 0.7-0.8),不达标走 fallback
  5. 降级契约:Seedance audio 不可达时 cloud-production 回退到无 audio 模式(generation 不阻断),degraded E2E 仍 <5s,degrade 路径有单元测试覆盖(DEGRADE-01/02)
**Plans**: TBD
**UI hint**: yes

### Phase 23: CreativeHistoryTracker
**Goal**: 旗舰能力上线 — 每镜头 creative_history 可追溯,改剧本一行自动定位受影响镜头列表(Git-for-AIGC-movies MVP)
**Depends on**: Phase 20
**Requirements**: B4-01, B4-02, B4-03, B4-04, B4-05, B4-06
**Success Criteria** (what must be TRUE):
  1. AssetBus envelope 格式落地(`{value, derived_from, content_hash}`),向后兼容 v2.0 数据可读
  2. `lib/creative-history-tracker.js` 实现 adjacency-list DAG + reverse BFS diff — 输入新 script_hash,输出受影响 shot_id 列表(24 集规模 <500ms)
  3. Blast radius cap 生效(默认 ≤20 shots,超出时提示 operator scope,非静默全量重渲),依赖图 lazy on-demand 遍历(无 dependency explosion)
  4. Hash-stamping 嵌入 cloud-production 下游 lineage MVP(每个生成视频带 `{shot_id, source_hash}`);上游 script→sts→shot retrofit 明确标记为 v3.1 out-of-scope
  5. 降级契约:CreativeHistoryTracker append 失败时 fire-and-forget(不阻断 pipeline),degraded E2E 仍 <5s,单元测试覆盖 AssetBus 不可达路径(DEGRADE-01/02)
**Plans**: TBD

### Phase 24: CrossEpisodeAssetIndex
**Goal**: 跨剧集角色资产复用 — 同主角系列剧第二集起,L1/L2 阶段命中 library 复用资产(研究级匹配精度 + human gate)
**Depends on**: Phase 20 (并行 track,独立于 Phase 21/22/23)
**Requirements**: B2-01, B2-02, B2-03, B2-04, B2-05, B2-06
**Success Criteria** (what must be TRUE):
  1. `_computeCostumeFingerprint` 从 SHA-256(paths) 重写为 DINOv2 embedding(主)+ pHash(降级)双索引;`lib/perceptual-hash.js`(DCT-II,~80 LOC,零 npm 依赖)实现
  2. `CharacterAssetManager.findByIdentity(fingerprint, threshold)` 跨 episode 查询接口可用,根路径约定 `projects/.shared/character-library/`
  3. 两阶段匹配落地:hash 检索 → DINOv2 cosine ≥0.92 确认(单独 pHash 命中不写入 library,避免 Pitfalls 陷阱 3 误匹配)
  4. 50+50 pair 标注评估集(同演员不同角色 / 同角色跨剧集)产出,threshold 经验校准报告落地,首次跨 episode 匹配走 human gate(operator 审批后才写入 library)
  5. 降级契约:DINOv2 embedding 不可达时降级到 pHash-only(精度下降但功能可用),degraded E2E 仍 <5s,两条匹配路径各有单元测试(DEGRADE-01/02)
**Plans**: TBD

### Phase 25: FineTuningETL
**Goal**: 数据回流闭环就绪 — failed case → JSONL manifest → operator 审批 → LoRA training 任务提交,含完整 launch-blocker 安全机制
**Depends on**: Phase 20, Phase 21 (推荐也等 Phase 22/23 最大化数据价值,但硬依赖只到 21)
**Requirements**: B6-01, B6-02, B6-03, B6-04, B6-05, B6-06
**Success Criteria** (what must be TRUE):
  1. `lib/finetune-etl.js` 消费 Hermes audit + failed_shots 产出 JSONL manifest,每条样本含 `(failed_shot, anchor, audio, recommended_action)`
  2. LoRA training 任务通过 `gtClient.submitTask({task_type: 'lora_training', ...})` 提交接口可用(v3.0 只产 manifest + 提交能力,实际训练由 operator 触发)
  3. **Human review gate 作为 launch blocker** — 每条样本必须含 `{copyright_status, pii_scrubbed, label_correct, approved_for_training}` 4 字段,未经 operator 审批的样本不可进入训练集(强制阻断,非提示)
  4. PII scrubber 检测并脱敏训练数据隐私内容;Golden-set regression test 训练前后跑 baseline(50-100 known-good prompts)确保不破坏 v2.0 能力;Dataset poisoning 检测(基于 SilentBadDiffusion 论文,异常样本 outlier 检测)三项安全机制全部就位
  5. 降级契约:FineTuningETL 是 async queue(Hermes 不可达时 best-effort enrichment,不阻断 manifest 产出),degraded E2E 仍 <5s,单元测试覆盖 Hermes / gold-team 不可达路径(DEGRADE-01/02)
**Plans**: TBD
