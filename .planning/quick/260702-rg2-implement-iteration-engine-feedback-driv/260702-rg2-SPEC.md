# Iteration Engine — 基于反馈的版本化迭代引擎

## 问题陈述

kais-movie-agent 的无限画布已有完整的反馈采集（FeedbackPanel / ReviewActionButtons）、分支模型（FlowBranch）、节点执行（gold-team engine）、拓扑传播（feedback propagation）能力。但缺少一个**迭代编排层**：当用户批量审阅给出反馈后，系统无法智能诊断问题根因并自动生成下一版资产。

用户需求：
1. 批量审阅画布节点后，系统综合所有反馈做**智能诊断**
2. 区分三种问题类型：抽卡问题（重跑即可）/ 系统性缺陷（改管线再重跑）/ 上游污染（回退上游）
3. 自动 fork 分支、重新生成资产、挂到新分支
4. 用户 approve 新版 → 分支转正；reject → 丢弃分支回退
5. 如果诊断认为管线需要调整，需用户确认后才修改

## 与 PipelineReflector 的关系

- **PipelineReflector**（已有）：跨集策略优化，从驳回模式中提取教训改 formula eval_score
- **IterationEngine**（本次）：单集内版本化迭代，诊断具体反馈 → 重新生成资产 → 版本对比
- 两者互补：IterationEngine 做完一轮迭代后，可以把"什么有效什么无效"喂给 Reflector 做跨集学习

### IterationPlan 数据结构

```typescript
interface IterationPlan {
  id: string;                          // "iter-{timestamp}-{random}"
  episodeId: string;
  branchLabel: string;                 // "v2-表情优化"
  diagnosis: {
    type: "reroll" | "pipeline_adjust" | "upstream_fix";
    //   reroll          = 抽卡问题，重跑即可
    //   pipeline_adjust = 系统性缺陷，需改管线
    //   upstream_fix    = 上游污染，需回退上游
    rootCause: string;                 // 人话描述根因
    confidence: number;                // 0-1
    evidence: string[];                // 支撑判断的证据
  };
  actions: IterationAction[];
  requiresApproval: boolean;           // 如果有 pipeline_adjust 则 true
  createdAt: string;
}

interface IterationAction {
  nodeId: string;                      // 要重新生成的节点
  action: "regenerate" | "regenerate_after_parent" | "skip";
  promptDelta?: string;                // prompt 增补（LLM 建议的改进词）
  pipelineAdjustment?: {               // 仅 pipeline_adjust 类型
    type: "prompt_modification" | "threshold_adjustment" | "parameter_change";
    target: string;
    change: string;
  };
  reason: string;
  dependsOn?: string[];                // 拓扑依赖（上游先跑完）
}

interface IterationResult {
  planId: string;
  branchId: string;                    // 新创建的分支 ID
  regeneratedNodes: Array<{
    nodeId: string;
    newNodeId: string;                 // 新分支下的节点 ID
    status: "success" | "failed" | "pending";
    outputUrl?: string;
  }>;
}
```

### 诊断 LLM Prompt 模板

```
你是一个 AI 短剧创作管线的迭代诊断专家。
你的任务是根据用户对一批资产的反馈，诊断问题根因，制定下一版迭代计划。

## 反馈数据
{feedback_by_node}

## 节点拓扑关系
{topology_summary}

## 历史评估数据
{evaluation_summary}

## 诊断框架

请判断问题属于以下哪一类：

### A. reroll（抽卡问题）
特征：个别镜头质量不稳，但同类镜头有成功的。问题出在随机性/运气。
判据：同一 phase 下大部分节点 score 正常，只有少数低分被 reject。
动作：只对被 reject 的节点换 prompt 微调 + 重新生成，不改管线。

### B. pipeline_adjust（系统性缺陷）
特征：某 phase 的节点大面积同类问题。问题出在 prompt/参数设计。
判据：多个 reject 原因指向同一模式（如"表情呆板"出现 5 次以上）。
动作：先修改管线参数 → 再重新生成受影响的所有节点。

### C. upstream_fix（上游污染）
特征：下游节点全崩，但根因在上游（如角色图质量差导致所有镜头都丑）。
判据：feedback propagation 显示下游 reject 聚集，且上游节点评分也偏低。
动作：回退到上游问题节点重新生成 → 下游级联重生成。
```

### API 设计

后端新增路由文件：`src/routes/v1/iteration/index.ts`（kais-aigc-platform 侧）

```
POST /api/v1/iteration/plan
  Body: { projectId, episodesId, workdir }
  → 收集反馈 → LLM诊断 → 返回 IterationPlan
  → 不执行任何生成，只返回计划供用户审阅

POST /api/v1/iteration/execute
  Body: { projectId, episodesId, workdir, planId }
  → 如果 plan.requiresApproval && 未确认 → 400 报错
  → fork分支 → 按拓扑序重新生成 → 返回 IterationResult
  → 长时间运行，通过 WebSocket 推送进度

POST /api/v1/iteration/confirm
  Body: { projectId, episodesId, branchId }
  → 用户 approve 新版 → branch.status=active, 旧分支 archived
  → 如果有 pipelineAdjustment → 写入 prompt-overrides.json

POST /api/v1/iteration/discard
  Body: { projectId, episodesId, branchId, reason? }
  → 用户 reject → branch.status=rejected
  → 自动回退到上一个 active 分支

GET /api/v1/iteration/plans
  Query: { projectId, episodesId }
  → 返回历史迭代计划列表

GET /api/v1/iteration/status/:planId
  → 返回迭代执行状态（进行中/完成/失败）

POST /api/v1/iteration/approve-adjustment
  Body: { planId }
  → 如果 plan 有 pipelineAdjustment，用户确认管线改动
  → 才能执行 execute
```

### 执行流程（execute 内部）

```javascript
async execute(plan) {
  // 1. Fork 分支
  const branchId = await this._forkBranch(plan);

  // 2. 如果有 pipeline adjustment 需要先应用
  if (plan.requiresApproval && !plan.adjustmentApproved) {
    throw new Error('Pipeline adjustment requires approval before execution');
  }
  if (plan.adjustmentApproved) {
    await this._applyPipelineAdjustment(plan);
  }

  // 3. 按拓扑序执行 actions
  //    先执行无 dependsOn 的，完成后执行依赖它们的
  const sorted = this._topologicalSort(plan.actions);

  const results = [];
  for (const action of sorted) {
    // 在新分支下克隆节点
    const newNodeId = `${action.nodeId}-v${plan.iterationVersion}`;

    // 构建 prompt（原 prompt + delta）
    const prompt = await this._buildPrompt(action);

    // 调用 gold-team 引擎
    const taskResult = await this._callEngine(action.nodeId, prompt, branchId);

    // 新节点挂到画布新分支下
    await this._createNodeOnBranch(newNodeId, action.nodeId, branchId, taskResult);

    // WebSocket 推送进度
    this._broadcast('iteration:progress', { planId, newNodeId, status: 'done' });

    results.push({ nodeId: action.nodeId, newNodeId, status: 'success', outputUrl: taskResult.outputUrl });
  }

  return { planId: plan.id, branchId, regeneratedNodes: results };
}
```

## 实现约束

1. **不自动执行** — plan 返回后必须用户确认才 execute
2. **不自动确认管线改动** — pipelineAdjustment 需要单独 approve
3. **分支隔离** — 新版资产挂在新分支下，不影响原有分支
4. **拓扑顺序** — 重生成必须按拓扑序（上游先跑完下游再跑）
5. **LLM 调用复用** — 用 `hermes-adapter.js` 的 `callLLM()`
6. **ES Module** — 项目 `"type": "module"`
7. **零外部依赖** — 只用 Node.js 内置 + 已有依赖
8. **失败降级** — gold-team 调用失败时标记该节点 failed，不中断整个迭代
9. **引擎调用通过 HTTP** — iteration-engine.js 在 kais-movie-agent 侧（Node.js ESM），需要调用 kais-aigc-platform 的画布 API 和 gold-team engine。通过 HTTP fetch 调用 `http://localhost:10588/api/...`，不直连数据库
10. **WebSocket 实时推送** — 迭代进度通过 `broadcastToProject` 推送到画布前端
