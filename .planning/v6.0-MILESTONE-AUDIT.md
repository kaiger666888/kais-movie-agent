## Warnings (Non-Blocking Tech Debt)

### W-1: p10b Outputs Not Consumed by Phase 41/42 (Intentional, v7.0+ Deferred)

**What:** Phase 40's `rapid-preview-clips` JSONL and `episode-meta` JSON slots are write-only within the pipeline — no Phase 41/42 module reads them.

**Why this is OK:** Documented as intentional operator-side consumption per Phase 40 CONTEXT (`reader_phases: []`, "consumed operator-side per blueprint Out of Scope"). Phase 41 CONTEXT explicitly defers "Recipe auto-application to p10b" to v7.0+ (REQUIREMENTS.md backlog section A).

**What convergence loop DOES close in code:** `story-framework/final-audit → recipe → feedback → Wilson CI` — fully wired and tested via 10 E2E integration tests.

**v7.0+ remediation:** When ready to auto-promote winning preview variants, add a `recipe_library.capture_from_preview_clips(episode_id)` helper that reads `rapid-preview-clips` slot. Until then, operators consume preview data via CLI/REPL.

### W-2: Documentation Drift — Phase 40 4-param vs Phase 41 5-param structure{}

**What:** Phase 40 CONTEXT.md claims p10b's 4 STRUCTURE_PARAMS (`hook_position_sec / emotion_sequence / turning_points_sec / ending_state`) "align 1:1 with Phase 41 emotion-recipe `structure{}` fields". Phase 41's `_validate_structure` actually requires 5 fields (adds `emotion_drop_level`).

**Why this is conceptually correct:** `emotion_drop_level` is derived from `final-audit.scores.D2_emotion` (a quality outcome), not a creative knob p10b should A/B test. The 4-vs-5 split is intentional — p10b varies what operators can tune; the recipe captures both tunable params AND quality-derived fields.

**Remediation:** Documentation-only. Update Phase 40 CONTEXT.md prose to clarify: "p10b varies the 4 tunable structure params; recipe stores 5 fields including emotion_drop_level which is derived from final-audit quality scores, not varied by p10b." Optional cleanup; no REQ impact.

### W-3: E2E Test Does Not Assert converged=True

**What:** Phase 42 E2E Test 2 ("Convergence after 10 feedbacks") asserts `sample_size >= 10` and CI format, but does NOT assert `converged=True` end-to-end. The convergence threshold logic (`_is_converged` at `recipe_library.py:348-361`) is verified at the unit level only.

**Why this is OK in practice:** Wilson CI at `sample_size=10` with `completion_rate=0.5` produces a ±13% spread — mathematically NOT converged. Convergence genuinely requires ~1000 samples. Unit Test 14 in Phase 41 verifies the threshold works at scale (Test fixture: 500/1000 → converged=true). The E2E test correctly does NOT lie about convergence at small sample sizes.

**Remediation:** Optional — add an E2E test that runs 1000 feedbacks and asserts `converged=True`. Marginal value given unit coverage exists. Defer to v6.1.
