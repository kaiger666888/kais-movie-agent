**Auditor:** Claude Code (cross-phase integration auditor)

**Milestone scope:** Phases 10–17 (8 phases, 19 REQ-IDs across ARCH/QUAL/PERF/SAFE/E2E)

## 1. Executive Summary

The v2.0 milestone successfully landed the **P0 architecture remediation**: PHASES array is 100% aligned with `phaseHandlers` (25 handlers cover all 20 V6 phases), HermesClient no longer rejects V6 phases, quality-gate no longer fakes an 80% pass, CompositionEngine is shell-injection-safe, and the pipeline demonstrably runs end-to-end in degraded mode (151/151 tests pass, including a 7-assertion E2E suite).

However, two structural gaps remain between the milestone's literal success criteria and what was actually delivered:

1. **12 of 15 V6 handlers are still Phase-10 stubs** (only `consistency-guard`, `character-generation`, `cloud-production`, `delivery` were replaced with real bodies; `consistency-guard` is partially real). `pain-discovery` → `script-lock` and `final-audio` still write `{_stub: true}` JSON with empty arrays. This is **not** a regression — Phase 10's plan was explicit that these are stubs — but it means many REQ-IDs are met at the "routing works" level, not the "real content produced" level.

2. **E2E-01/E2E-02 success criteria are not literally satisfied.** The criteria require "final.mp4 exists, playable, ~60s, with wav audio track." Phase 17 ships a Layer-1 degraded-mode test that verifies JSON artifacts exist and the pipeline exits 0, but explicitly **does not produce final.mp4**. Layer-2 (real GPU run) is documented in `docs/E2E-RUNBOOK.md` but not executed. The Phase 17 CONTEXT.md itself acknowledges: "本 phase 不强求产出真实视频."

**Recommendation:** **Ship v2.0 as an internal milestone** (architecture remediation complete, regression-safe), but **do not claim production E2E validation** until a Layer-2 real-GPU run produces a playable `final.mp4`. Treat "real content for upper-half creative phases (pain→script)" and "real-service final.mp4" as the top of the v3.0 backlog.

### Missing or Fragile Connections

#### WARNING W-1: 12 V6 handlers remain Phase-10 stubs

`grep -oE "_pendingRealImplementation: 'phase-[0-9]+'" lib/phases/index.js | sort | uniq -c` returns:
- **15 × `phase-11`** — pain-discovery, topic-selection, outline-generation/selection, script-generation/selection, script-lock, soul-visual, soul-voice, geometry-bed (creative上半段, never实化 in any v2.0 phase)
- **6 × `phase-14`** — character-selection, scene-generation, scene-selection (Phase 14 only实化'd character-generation itself)
- **2 × `phase-13`** — delivery (Phase 13 only hardened quality-gate.js, did NOT replace delivery handler body — `_stub: true` still present at `:2610`)
- **2 × `phase-15`** — final-audio (Phase 15 added parallelism scaffold but final-audio handler body still writes `_stub: true` at `:2538` with `_pendingRealImplementation: 'phase-15'`)

**Impact:** In degraded E2E, this is fine (handlers degrade gracefully). In a real Layer-2 GPU run, the upper creative half (pain→script) will produce empty `script-candidates.json: {candidates: []}` and downstream phases will have no real scripts/scenes to render. The pipeline will complete but produce nothing meaningful.

**Severity:** WARNING — Phase 10's design was intentionally stub-first; this is documented tech debt, not a hidden bug. But it limits the practical value of any Layer-2 E2E attempt.

#### BLOCKER B-1: E2E-02 success criterion literally requires `final.mp4` — no test produces one

**Evidence:**
- REQUIREMENTS.md E2E-02: "`projects/<new-project>/final.mp4` 存在、可播放、时长 ≈ 60s,且包含 wav 音轨"
- Phase 17 VERIFICATION marks E2E-02 as ✅ with rationale "produces consistency-pass.json / cost-report.json / quality-report.json" — **none of which is final.mp4**
- Phase 17 CONTEXT.md line 17: "本 phase 不强求产出真实视频(需 gold-team GPU)"
- `composition` handler at `:1355` does call `composer.compose({ outputPath: 'final.mp4' })`, but with stub `video_tasks.json` input, compose() fails or no-ops in degraded mode
- No CI test asserts `final.mp4` exists or is playable

**Severity:** BLOCKER for "ship-ready" claim, **acceptable** for "internal milestone" claim. Phase 17 honestly scoped itself as Layer-1 (degraded) + Layer-2 (documented), but REQUIREMENTS.md E2E-02 was not amended to match. The traceability table and the verifications are out of sync with the literal SC wording.

#### WARNING W-2: QUAL-01 `_llmIdentityScore` fallback returns 0.7

**Evidence:** `lib/continuity-auditor.js:487-489` — when `_getDINOv2Score` returns null (no anchors) and the textual fallback `_llmIdentityScore` is tried, LLM failure returns `return 0.7` (the lower threshold). This is the textual-fallback path, not the image-scorer path; `_getDINOv2Score` itself correctly returns null on failure.

**Impact:** Minor — in the no-anchors + LLM-down scenario, identity_match gets 0.7 (passes threshold) instead of null (unscored). This is a "fail-open with borderline pass" not a "fail-fake" — but it does mean an asset could pass consistency audit without any real evaluation.

**Severity:** WARNING — Phase 12's VERIFICATION claims "Failure returns null (not 0.7)" but this is only true for the image-pair path. The textual fallback still has a 0.7 default.

#### WARNING W-3: No real GLM-4V API test

Phase 12 VERIFICATION explicitly notes tests run against expired/missing `ZHIPU_API_KEY` and rely on 401 responses as proof of "real call." This proves wiring but does not prove scoring quality. The cache layer masks repeated failures, so a stale 401 could persist undetected.

**Severity:** WARNING — operator must set a valid `ZHIPU_API_KEY` before Layer-2 E2E; documented in runbook.

### Overall status
**`gaps_found` + `tech_debt`** — the architectural bleeding is stopped (P0 issues all fixed), but the patient is not yet running marathons (real content + real GPU E2E unproven). Ship as internal milestone; do not claim "first 60s short film produced" until Layer-2 E2E is executed.
