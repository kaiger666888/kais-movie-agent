<objective>
Verify via integration tests that the hermes-agent `PluginManager` discovers all 3 new plugins (from Plan 31-01) in the bundled-plugins scan, parses their manifests, and can load them when explicitly enabled via `plugins.enabled` config.

Purpose: Plan 31-01 proved each plugin's `register(ctx)` works in isolation with a fake ctx. This plan proves the **actual hermes-agent loader** can find, parse, and execute those registrations end-to-end. ROADMAP SC#2 demands "hermes-agent plugin loader 启动时三个插件注册成功(无 import error,日志可见 'plugin loaded')". This is the plan that satisfies SC#2.

**Key insight from loader code (`hermes_cli/plugins.py:1278-1301`):** bundled `kind: standalone` plugins are **opt-in** — they're discovered (manifest parsed, recorded in `list_plugins()`), but they only load (i.e. execute `register(ctx)`) when listed in `plugins.enabled` config. The tests must cover BOTH states:
1. Discovery: plugin appears in `list_plugins()` with `enabled=False` and `error="not enabled in config ..."` — this proves the manifest was found and parsed.
2. Enabled load: when config sets `plugins.enabled: [kais_aigc, ...]`, the plugin loads, `register(ctx)` runs, and `tools_registered` count is 4.

This is why the phase_context chose `kind: standalone` — the plugins don't auto-load into every hermes-agent session (which would disrupt users not yet on v5.0), but they're discoverable and loadable on demand.

Output: 6 test files (one `tests/__init__.py` + one `tests/test_loader_discovery.py` per plugin).
</objective>

```python
def get_plugin_manager() -> PluginManager: ...   # singleton accessor
def discover_plugins(force: bool = False) -> None: ...  # triggers scan

class PluginManager:
    def discover_and_load(self, force: bool = False) -> None: ...
    def list_plugins(self) -> List[Dict[str, Any]]: ...
    # Each list_plugins() entry: {name, key, kind, version, description,
    #   source, enabled, tools, hooks, middleware, commands, error}

def _get_enabled_plugins() -> Optional[set]: ...  # reads plugins.enabled from config.yaml
def _get_disabled_plugins() -> set: ...           # reads plugins.disabled from config.yaml
```

Config contract (config.yaml):
```yaml
plugins:
  enabled:    # opt-in allow-list. Plugin names OR path-derived keys (e.g. "kais_aigc")
    - kais_aigc
    - pipeline_state
    - review_gates
  disabled: []  # explicit deny-list, always wins over enabled
```

Test isolation strategy:
- Each test must call `get_plugin_manager().discover_and_load(force=True)` to trigger a fresh scan.
- Mutating config mid-test requires monkeypatching `hermes_cli.plugins._get_enabled_plugins` to return the desired set (avoids touching the real `~/.hermes/config.yaml`).
- Reset state between tests by calling `discover_and_load(force=True)` again with different enabled sets.
- Tests MUST NOT pollute the global `tools.registry` permanently — use the enabled/disabled gating so register() only runs when explicitly enabled, OR assert and clean up. (Loader's force=True clears all plugin state including registered tools from prior loads.)
