<objective>
Implement `canvas.py` — the Python port of `lib/canvas-client.js` (HTTP-only subset) + `lib/canvas-content-sync.js` saveGraph helper. Wave 1 plan, fully parallel with 32-01/02/04.

**Key v5.0 hardening (CRITICAL-FINDING-05 + D-08):**
- HTTP API v2 ONLY — no sqlite. The Node.js `canvas-content-sync.js` still reads via `sqlite3` CLI (legacy), but the Python port uses HTTP `load-v2` for both reads and writes. This is the v5.0 standardization (CANVAS-IN-HERMES-03).
- The Python client exposes `save_canvas(graph)` and `load_canvas()` ONLY. The rich Node.js node/edge/branch/variant operations (`addNode`, `addLink`, `createBranch`, `patchCanvas`, etc.) are out of scope — orchestration (Phase 35+) calls `save_canvas(merged_graph)` after building the graph in Python. Specialized operations belong to higher layers.
- WebSocket support (`socket.io-client` in Node.js) is NOT ported — Phase 32 is sync HTTP only. WS comes back if Phase 37+ needs live canvas events.

Output: 2 files (client module + test file). ~120-200 LOC client + 6-8 tests.

**Why smallest of the 4:** Canvas API surface in Python is minimal (save + load + degrade). The Node.js client has ~850 LOC because of WS + node ops + branch ops + variant ops; the Python port intentionally trims to the 2 operations that the v5.0 pipeline actually needs.
</objective>
