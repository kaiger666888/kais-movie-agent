**Rewrite 4 handler bodies:**

1. **`_handle_gate_submit(args, **kw) -> str`**:
   - Extract `gate_id`, `episode_id`, `mode` (optional, default None → use gate's default_mode), `payload` (default {}).
   - Validate `gate_id` + `episode_id` required (else `tool_error`).
   - Look up gate config; if `gate_id` not in `GATE_REGISTRY` → `tool_error("Unknown gate_id: ...")`.
   - Call `pause_for_review(gate_id, episode_id, payload, mode=mode)`.
   - On `GateMaxRetriesExceeded`: call `mark_episode_failed(episode_id, gate_id, exc)` then return `tool_error({"status": "episode_failed", "gate_id": ..., "error": str(exc)})`.
   - On success: return `tool_result({"status": "submitted", "gate_id", "episode_id", "review_id", "attempt"})`.

2. **`_handle_gate_wait(args, **kw) -> str`**:
   - Extract `gate_id`, `timeout_sec` (default 3600).
   - Validate `gate_id` required.
   - Look up gate; if unknown → tool_error.
   - This is a thin shim — Phase 34 doesn't expose raw `gate.wait()` to LLM (blocking inside a tool call is dangerous). Instead, return `{"status": "use_resume_from_callback", "gate_id": ..., "instructions": "For webhook mode, POST HMAC callback to resume; for blocking mode, call gate_resolve directly"}`.
   - **Alternative (pick one):** delegate to runner_hooks.poll_until_terminal for polling mode only; for blocking/webhook, return guidance to use gate_resolve. Document the choice in handler docstring.

3. **`_handle_gate_resolve(args, **kw) -> str`**:
   - Extract `gate_id`, `decision` (required, enum approve/reject/contest), `suggested_action` (optional).
   - Validate required fields.
   - Construct a synthetic callback body: `{"gate_id": gate_id, "decision": decision, "suggested_action": suggested_action}`. Call `resume_from_callback(body=json.dumps(...), signature="", timestamp=int(time.time()))`.
   - **Note:** the HMAC verify path in `resume_from_callback` will reject empty signature if a callback secret is set. For direct operator invocation (blocking mode resume), the operator is already authenticated to hermes-agent; we trust the call. Either:
     - (a) Add a `skip_hmac=True` parameter to `resume_from_callback` for direct invocation, OR
     - (b) Add a separate `resolve_direct(gate_id, decision, suggested_action)` function in runner_hooks that bypasses HMAC (used by this handler), keeping `resume_from_callback` strict for external callbacks.
   - **Recommendation:** Option (b) — cleaner separation. Add `resolve_direct()` to runner_hooks in this plan (small addition; update 34-03 tests is not required since it's a new function).
   - Return `tool_result({"status": "resolved", "gate_id", "decision", "suggested_action", "rollback_to": ...})`.
   - On `GateError` (invalid decision, no pending gate) → `tool_error`.

4. **`_handle_gates_list(args, **kw) -> str`**:
   - Extract optional `episode_id` (filter — Phase 34 scope: ignore for now, return all 8).
   - Return `tool_result({"gates": [...8 gate configs as dicts...], "count": 8})`.
   - Use `GATE_REGISTRY` directly (already loaded at import).
